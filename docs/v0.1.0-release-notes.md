# v0.1.0 Public Preview Release Notes

Schema Docs v0.1.0 is a public preview of a local-first AI document intake layer. It is built for people who already send documents, tables, and reports into AI tools and want to preview, mask, trim, block, and audit that context before anything leaves the machine.

The visible entry is Office, PDF, spreadsheet, CSV, TXT, and Markdown files. Markdown is the technical exchange layer behind AI preview, API routes, audit records, and SDXP handoff packages.

## Who This Preview Is For

- People already using ChatGPT, Claude, Gemini, local models, or private LLM workflows with documents.
- Teams that need a local check before employees upload files or paste document content into AI tools.
- Developers and AI workflow builders who want an API-first document intake layer with review evidence.
- Researchers, analysts, writers, legal, finance, and consulting users who need clean AI context from messy documents.

## What Works

- Create a local workspace with a manifest.
- Create a temporary local workspace from the Web/Desktop UI without first typing a path.
- Select a workspace folder from the Desktop UI through a Windows native folder picker.
- Run a visible first-workflow check from the Web/Desktop UI to generate an Office-to-Markdown-exchange DOCX/PDF/package read-back loop.
- Generate and import a sample Word document from the Web/Desktop UI for a no-file first workflow.
- Select an existing local file from the Desktop UI through a Windows native file picker, then import it through the same local API path.
- Import supported local files into workspace records.
- Convert supported DOCX, PDF text-layer, TXT, and Markdown inputs through a Markdown exchange-layer flow.
- Write separate AI-ready Markdown and human-readable Markdown outputs, with automatic numbered human-readable part files for long documents.
- Export Markdown to DOCX and PDF.
- Create directory-based Markdown Exchange Packages with `document.md`, `document.schema.json`, `manifest.json`, `evidence.jsonl`, `tables/`, `assets/`, and `exports/`.
- Read back and hash-verify exchange packages.
- Generate receiver-facing exchange package trust artifacts: `receiver-report.md` for human review and `trust-report.json` for automation.
- Preview exactly what AI will see before sending.
- Block confirmed AI send when Send Gate returns `review_required`.
- Prepare filtered table context locally before AI review.
- Write AI responses back into Markdown exchange records.
- Save body-free runbooks for long-document feeding where each batch still requires review.
- Open long-document human Markdown through a readable index and clickable numbered part files instead of loading the full file into the editor by default.
- Save API profiles without persisting API keys.
- Use CLI, local HTTP API, local SDK, smoke flow, and Web UI prototype.
- Start from a Tauri v2 desktop shell scaffold.
- Produce Windows MSI/NSIS bundles as packaging proof.
- Include a desktop runtime bridge that starts the bundled JS runtime through packaged `runtime/node.exe` when present, with system `node` as fallback.
- Show Desktop runtime diagnostics for Node availability, runtime resource paths, session logs, and local API health.
- Keep strict desktop verification as the final manual release gate for the current Windows artifact set, including visible diagnostics, visible first workflow, native workspace/file pickers, WebView2 confirmation, runtime diagnostics, Send Gate evidence, and release artifact SHA-256 match.

## What This Release Is Not

- It is not a complete Office compatibility layer.
- It is not a high-fidelity PDF layout engine.
- It is not a high-accuracy handwriting or mathematical OCR product; the optional local Tesseract + Poppler path still requires visual review.
- It is not a cloud document sync service.
- It is not a general document management system like Notion, Obsidian, or Typora.
- It does not automatically send documents or workspace files to a model provider.

## Privacy And Trust Boundary

- Workspace files stay local unless a user explicitly chooses a workflow that sends content.
- AI preview is dry-run and does not send network requests.
- Confirmed AI send requires explicit confirmation.
- API keys are not stored in API profiles.
- Audits and evidence records store hashes, lengths, policy decisions, and summaries rather than API keys or full prompt bodies by default.
- Exchange package read-back verifies declared file hashes and rejects tampered declared files.

## Known Limits

- DOCX import currently reads `word/document.xml` only.
- DOCX styling and layout fidelity are minimal.
- Markdown to DOCX is a polished structured export, HTML is a printable styled export, and PDF uses the same offline HTML/KaTeX pipeline for A4 output with rendered formulas and embedded local images. PDF export requires Edge or Chromium and now reports a clear error instead of silently producing a degraded text-only file.
- PDF extraction works best with text layers. Scanned PDFs can use optional local Tesseract + Poppler OCR; without those tools the workflow remains blocked, and handwriting, formulas, tables, and reading order require visual review.
- XLSX support reads the first sheet only and does not execute formulas.
- The query engine is a small SQL subset, not DuckDB.
- The Web UI can create a temporary workspace and sample Word document, but selecting arbitrary existing local folders/files from a browser still requires pasted paths. The Desktop UI has Windows native workspace and file pickers.
- Desktop shell is present as a packaging path and the current Windows artifacts have strict verification evidence, but a broader clean-machine test matrix is still needed before calling the Windows app polished.
- Production desktop bundles now include a packaged Node runtime resource, but broader clean-machine installer testing is still required before calling the Windows app polished; see `docs/desktop-runtime-gap.md` and `docs/desktop-standalone-decision.md`.
